GPU Server 端：開放防火牆 Port 在 GPU Server 上操作

確保開發機能夠連到 LiteLLM Proxy 的 Port 4000：

Ubuntu / Debian（ufw） 
# 開放 RAG Proxy port（Claude Code 連這裡）
sudo ufw allow 4001/tcp comment 'RAG Proxy'

# 也開放 LiteLLM 直連（可選，用於不經 RAG 的情境）
sudo ufw allow 4000/tcp comment 'LiteLLM Proxy'

# Open WebUI（可選）
sudo ufw allow 8080/tcp comment 'Open WebUI'

sudo ufw reload
CentOS / RHEL（firewalld） 
sudo firewall-cmd --permanent --add-port=4001/tcp
sudo firewall-cmd --permanent --add-port=4000/tcp
sudo firewall-cmd --permanent --add-port=8080/tcp
sudo firewall-cmd --reload
無防火牆 / 已由路由器管理

如果 Server 在同一區網且無防火牆限制，可跳過此步驟。

GPU Server 端：設定 API Key（多人隔離）在 GPU Server 上操作

每位團隊成員需要一把專屬 API Key 才能使用 RAG Proxy。Key 綁定身份和預設專案，實現身份驗證 + Per-project RAG 隔離 + GPU 排隊追蹤。

Step 1：為每位成員產生 Key

# 在 GPU Server 上產生隨機 Key
python3 -c "import secrets; print(f'rag-{secrets.token_urlsafe(32)}')"

# 範例輸出：rag-A7kX9mP2vQ8nR4wL6yT1bF3jH5sD0eK...
GPU Server Terminal

Step 2：編輯 API Keys 設定檔

編輯 config/api_keys.json，為每位成員加入一筆記錄：

{
  "keys": {
    "rag-Willis的Key": {
      "user_id": "willis",
      "project_id": "xse-apps",
      "enabled": true
    },
    "rag-Dev2的Key": {
      "user_id": "dev2",
      "project_id": "xse-api",
      "enabled": true
    },
    "rag-Dev3的Key": {
      "user_id": "dev3",
      "project_id": "xse-apps",
      "enabled": true
    }
  }
}
config/api_keys.json（GPU Server）
Key Config 欄位說明
欄位說明
user_id用於 log 追蹤和 GPU 排隊識別（顯示在 rag-proxy log 中）
project_id預設 RAG collection，自動路由到 codebase-{project_id}
enabled設 false 可立即停用該 Key（不需重啟服務）
Hot Reload：修改 api_keys.json 不需要重啟
RAG Proxy 會自動偵測檔案變更並重新載入 Key 清單。新增 / 停用 Key 即時生效。

開發機端：設定 Claude Code 環境變數 在開發機上操作

這是最關鍵的步驟 — 告訴你的 VSCode Claude Code 把 API 請求導向遠端 GPU Server 上的 RAG Proxy，並帶上你的專屬 API Key。

方法 A：Shell 環境變數 + API Key Header（推薦）

在開發機的 ~/.zshrc（macOS）或 ~/.bashrc（Linux）加入：

# ── Local LLM Stack（跨主機串接 + RAG + 多人隔離）──
# 將 GPU_SERVER_IP 替換為你的 GPU Server 實際 IP
# Port 4001 = RAG Proxy（驗證 + 排隊 + 向量搜尋）
export ANTHROPIC_BASE_URL="http://GPU_SERVER_IP:4001"
export ANTHROPIC_API_KEY="YOUR_LITELLM_MASTER_KEY"
~/.zshrc（開發機）

接著在 VSCode settings.json 中加入你的 API Key（按 Cmd+Shift+P → "Preferences: Open User Settings (JSON)"）：

{
  // 你的專屬 API Key（向 Server 管理員索取）
  "claude-code.apiHeaders": {
    "X-API-Key": "rag-你的專屬Key"
  }
}
settings.json（開發機 VSCode）

套用環境變數：

source ~/.zshrc
範例：假設 Server IP 是 192.168.1.100
# ~/.zshrc
export ANTHROPIC_BASE_URL="http://192.168.1.100:4001"
export ANTHROPIC_API_KEY="sk-local-dev-2025"

# VSCode settings.json
{
  "claude-code.apiHeaders": {
    "X-API-Key": "rag-A7kX9mP2vQ8nR4wL6yT1bF3jH5sD0eK"
  }
}
ANTHROPIC_API_KEY vs X-API-Key 差異
變數用途來源
ANTHROPIC_API_KEYLiteLLM Proxy 認證Server .env 的 LITELLM_MASTER_KEY（團隊共用）
X-API-KeyRAG Proxy 身份驗證 + 專案路由Server config/api_keys.json（每人不同）

臨時切換專案的 RAG Collection

你的 API Key 綁定了預設專案（如 xse-apps）。若需要臨時切換到其他專案的 RAG 資料，加上 X-Project-Id header：

{
  "claude-code.apiHeaders": {
    "X-API-Key": "rag-你的專屬Key",
    "X-Project-Id": "xse-api"   // 覆蓋 Key 預設的 project_id
  }
}
settings.json（開發機 VSCode）
不加 X-Project-Id 也完全沒問題
RAG Proxy 會自動使用你 API Key 綁定的預設 project_id。只有需要跨專案工作時才需要手動覆蓋。

啟用 RAG 向量搜尋（提升回應品質）在 GPU Server 上操作

Stack 內建 Qdrant 向量資料庫 + RAG Proxy，能將你的 codebase 建立索引，讓本地模型在回答時自動參考相關程式碼片段。

RAG 如何運作？
Claude Code 發送請求 → RAG Proxy 攔截 → 從問題中擷取語意 → 在 Qdrant 中搜尋相關程式碼 → 將檢索結果注入 system prompt → 轉發給 LiteLLM → 模型帶著上下文回答

Step 1：為每個專案建立獨立索引

使用 --project 參數為每個專案建立獨立的向量 collection，實現 Per-project RAG 隔離（在 GPU Server 上執行）：

# 為前端專案建立索引 → collection: codebase-xse-apps
./manage.sh index /path/to/xse-apps --project xse-apps

# 為後端專案建立索引 → collection: codebase-xse-api
./manage.sh index /path/to/xse-api --project xse-api

# 也可以直接用 indexer.py（在容器中或本機）
python3 services/rag-proxy/indexer.py /path/to/xse-apps --project xse-apps
GPU Server Terminal
Per-project Collection 命名規則
指令Qdrant Collection 名稱
--project xse-appscodebase-xse-apps
--project xse-apicodebase-xse-api
無 --projectcodebase（預設，向後相容）

重要：--project 名稱必須與 api_keys.json 中的 project_id 一致，RAG Proxy 才能自動路由到正確的 collection。

索引器會自動：

  • 掃描所有程式碼檔案（.py, .ts, .js, .vue, .go, .rs, .java 等 30+ 種）
  • 跳過 node_modules, .git, dist, build 等目錄
  • 將檔案切成 60 行一塊（10 行重疊）
  • 用 nomic-embed-text 模型生成向量嵌入
  • 存入 Qdrant 向量資料庫（per-project 獨立 collection）

Step 2：確認開發機環境變數

如果你已在 Step 6 中設定 ANTHROPIC_BASE_URL 指向 :4001（RAG Proxy），則 RAG 已自動啟用。確認一下：

# 確認指向 RAG Proxy（Port 4001，非 4000）
echo $ANTHROPIC_BASE_URL
# 應顯示：http://GPU_SERVER_IP:4001
開發機 Terminal
Port 差異一覽
Port服務用途
:4000LiteLLM（直連）不經過 RAG，直接路由到 Ollama
:4001RAG Proxy先搜尋 Qdrant，注入上下文後再轉發到 LiteLLM

:4000 vs :4001 使用情境比較

兩個 Port 各有適合的場景，可依需求自由選擇：

:4000 — LiteLLM 直連

  • 一般性程式問答：問語法、演算法、概念解釋等不需要專案上下文的問題
  • 快速回應優先：省去向量搜尋步驟，延遲更低、回應更快
  • 新專案 / 未建索引：尚未執行 ./manage.sh index 時，用 :4001 也沒有 RAG 效果
  • 多輪通用對話：debug 思路、架構討論、寫 README 等與特定檔案無關的任務
  • 節省 GPU 資源：不觸發 embedding 模型，減少 VRAM 消耗

:4001 — RAG Proxy（推薦）

  • 閱讀/修改專案程式碼：「這個函式做了什麼？」「怎麼改這段邏輯？」模型會自動參考相關檔案
  • 跨檔案理解：問到元件之間的依賴、API 呼叫鏈等需要上下文的問題
  • Code Review / 重構：RAG 注入相關程式碼片段，讓模型給出更精準的建議
  • 除錯排查：描述 bug 症狀，模型能搜尋到可能相關的程式碼區塊
  • 已建立索引的專案：執行過 ./manage.sh index 後效果最佳
簡單判斷原則
需要模型「知道你的程式碼」→ 用 :4001；只是「通用問答」→ 用 :4000 更快。
如果不確定，預設用 :4001 即可 — 即使沒有相關索引，RAG Proxy 也會正常轉發，只是不會額外注入上下文。

在 VSCode Claude Code 中自由切換 :4000 / :4001

Claude Code 透過環境變數 ANTHROPIC_BASE_URL 決定連線端點，切換方式非常簡單：

方法 1：Shell 別名快速切換（推薦）

在 ~/.zshrc 中加入以下 alias，之後用一行指令切換：

# ── LLM 模式切換 ─────────────────────────────────
# 切到 RAG 模式（:4001，帶驗證 + 排隊 + 向量搜尋）
alias llm-rag='export ANTHROPIC_BASE_URL="http://GPU_SERVER_IP:4001" && echo "已切換到 RAG 模式 (:4001)（需在 VSCode 設定 X-API-Key）"'

# 切到直連模式（:4000，更快，不經 RAG/驗證）
alias llm-direct='export ANTHROPIC_BASE_URL="http://GPU_SERVER_IP:4000" && echo "已切換到直連模式 (:4000)"'

# 切回 Anthropic 雲端
alias llm-cloud='unset ANTHROPIC_BASE_URL && export ANTHROPIC_API_KEY="sk-ant-你的Key" && echo "已切換到 Anthropic 雲端"'

# 查看目前模式
alias llm-mode='echo "ANTHROPIC_BASE_URL = ${ANTHROPIC_BASE_URL:-（未設定，使用 Anthropic 雲端）}"'
~/.zshrc（開發機）

使用方式：

# 切換後需要重新載入 VSCode 視窗才會生效
llm-rag       # 切到 RAG 模式
llm-direct    # 切到直連模式
llm-cloud     # 切回雲端
llm-mode      # 查看目前連到哪裡
開發機 Terminal
方法 2：VSCode Tasks 一鍵切換

在專案根目錄 .vscode/tasks.json 中加入切換任務，用 Cmd+Shift+P → "Tasks: Run Task" 快速切換：

{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "LLM: RAG 模式 (:4001)",
      "type": "shell",
      "command": "export ANTHROPIC_BASE_URL='http://GPU_SERVER_IP:4001' && echo '已設定 RAG 模式，請重新載入視窗 (Cmd+Shift+P > Reload Window)'"
    },
    {
      "label": "LLM: 直連模式 (:4000)",
      "type": "shell",
      "command": "export ANTHROPIC_BASE_URL='http://GPU_SERVER_IP:4000' && echo '已設定直連模式，請重新載入視窗 (Cmd+Shift+P > Reload Window)'"
    }
  ]
}
.vscode/tasks.json
切換後需要重新載入 VSCode
環境變數修改後，VSCode 的 Claude Code 擴充功能不會自動偵測變更。你需要：
1. 在 Terminal 執行 alias 指令（如 llm-rag）或 source ~/.zshrc
2. 在 VSCode 中按 Cmd+Shift+P → "Developer: Reload Window" 重新載入視窗
或者直接關閉再重開 VSCode，讓新的環境變數生效。

組織內部署通常不需要 HTTPS，直接用 HTTP 即可。若未來需要跨網段或對外開放，Stack 包含 Nginx SSL 反向代理：

# 1. 取消 docker-compose.yml 中 nginx 區塊的註解
# 2. 產生 SSL 憑證
./scripts/generate_ssl.sh
./manage.sh restart

然後在開發機改用 HTTPS 端點：

export ANTHROPIC_BASE_URL="https://GPU_SERVER_IP:443"
開發機 ~/.zshrc
自簽憑證注意
使用自簽憑證時，需在開發機設定環境變數 NODE_TLS_REJECT_UNAUTHORIZED=0，或將自簽 CA 加入系統信任清單。
使用固定 IP 或 Hostname

為避免 Server 重啟後 IP 變動，建議：

  • 方法 1：在路由器上為 GPU Server 設定 DHCP 保留（固定 IP）
  • 方法 2：在 Server 上設定靜態 IP
  • 方法 3：在開發機的 /etc/hosts 加入對應：
# /etc/hosts（開發機）
192.168.1.100    llm-server

之後環境變數可以寫成：

export ANTHROPIC_BASE_URL="http://llm-server:4000"
切換回 Anthropic 雲端 API 開發機端

只需移除或註解掉環境變數：

# 暫時切換回雲端
unset ANTHROPIC_BASE_URL
export ANTHROPIC_API_KEY="sk-ant-你的真正Anthropic-Key"

重啟 VSCode 後即會連到 Anthropic 雲端。想切回本地 Server，再 source ~/.zshrc 即可。

多人共用同一台 GPU Server（5+ 人團隊）

Stack 內建三層多人隔離機制，支援 5 人以上團隊同時使用：

1. API Key 驗證（身份識別）

  • 每位成員有專屬 X-API-Key，綁定 user_id 和預設 project_id
  • 沒有有效 Key 的請求會被 RAG Proxy 擋下（回傳 401）
  • 管理員可即時停用 Key（編輯 config/api_keys.json，設 enabled: false）
  • Hot Reload：修改 Key 檔案不需重啟服務

2. Per-project RAG Collection（上下文隔離）

  • 每個專案有獨立的 Qdrant 向量 collection（如 codebase-xse-apps、codebase-xse-api）
  • RAG Proxy 根據 API Key 的 project_id 自動路由到對應 collection
  • 成員可用 X-Project-Id header 臨時切換到其他專案的 collection
  • 不同專案的向量搜尋結果互不干擾

3. GPU 請求排隊（資源保護）

  • RAG Proxy 使用 Semaphore 限制同時送往 GPU 的推理請求數（預設 MAX_CONCURRENT_REQUESTS=3）
  • 超過上限的請求自動排隊等待，不會直接報錯
  • 排隊超過 QUEUE_TIMEOUT（預設 300 秒）才回傳 429
  • 可用 curl http://GPU_SERVER_IP:4001/queue/status 查看即時排隊狀態

管理員快速上手

# 為新成員產生 Key
python3 -c "import secrets; print(f'rag-{secrets.token_urlsafe(32)}')"

# 將 Key 加入 config/api_keys.json（不需重啟）
vim config/api_keys.json

# 為新專案建立 RAG 索引
./manage.sh index /path/to/new-project --project new-project

# 查看排隊狀態
curl http://localhost:4001/queue/status

# 查看誰在使用（log 中顯示 user_id）
./manage.sh logs rag-proxy | grep "[queue]"

常見問題排除

問題解決方式
connection refused / timeout 1. 確認 Server 服務在跑：docker ps
2. 確認防火牆開放 4000 port
3. 確認不是 localhost 而是 Server IP
Claude Code 顯示 API Key 無效 確認 ANTHROPIC_API_KEY 與 Server 上 .env 中的 LITELLM_MASTER_KEY 完全一致
401 authentication_error RAG Proxy 的 X-API-Key 驗證失敗。確認：
1. VSCode settings.json 中 claude-code.apiHeaders 有設 X-API-Key
2. Key 存在於 Server config/api_keys.json 且 enabled: true
429 rate_limit_error / GPU queue full GPU 排隊已滿且等待逾時。可能原因：
1. 太多人同時使用 — 稍等再試
2. 有人在跑大模型(70B)佔用 GPU 時間過長
3. 調高 MAX_CONCURRENT_REQUESTS 或 QUEUE_TIMEOUT
連線極慢 / 頻繁超時 1. 大模型首次載入 VRAM 需要時間
2. 檢查 Server 與開發機是否在同一子網
3. 嘗試改用較小模型（如 nemotron-nano）
CUDA out of memory 在 Server 上用 ./manage.sh status 查看 VRAM。減少 OLLAMA_MAX_LOADED_MODELS 或改用較小模型
Server IP 變動導致連不上 設定固定 IP 或使用 /etc/hosts hostname（見上方「使用固定 IP 或 Hostname」）
HTTPS 憑證錯誤（啟用 SSL 時） 內部部署建議直接用 HTTP。若已啟用 HTTPS，自簽憑證需設定 NODE_TLS_REJECT_UNAUTHORIZED=0
LiteLLM 報 "model not found" 在 Server 上確認模型已下載：./manage.sh list
回應品質不佳 本地模型能力有限，建議至少使用 nemotron-super (12B) 以上。啟用 RAG 可進一步改善
RAG 沒有注入 context 1. 確認已執行 ./manage.sh index 建立索引
2. 確認 ANTHROPIC_BASE_URL 指向 :4001 而非 :4000
3. 查看 ./manage.sh rag-status 確認 Qdrant 有資料
RAG Proxy 報錯 查看 ./manage.sh logs rag-proxy。常見原因：embedding 模型未下載（./manage.sh pull nomic-embed-text）
